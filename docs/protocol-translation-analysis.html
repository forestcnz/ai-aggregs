pub trait StreamConverter: Send {

// stream.rs:289
struct Chained<A: StreamConverter, B: StreamConverter>(A, B);

// stream.rs:321
fn make_converter(src: Protocol, dst: Protocol) -> Box<dyn StreamConverter> {
    match (src, dst) {
        (Protocol::Anthropic, Protocol::Responses) => Box::new(Chained(
            AnthropicToChatStream::new(),
            ChatToResponsesStream::new(),
        )),
        // ...
    }
}

设计亮点：通过泛型 Chained<A, B> 把"Anthropic → Responses"实现为"Anthropic → Chat → Responses"两段管线串联，避免为该方向单独写一遍状态机。这是静态分发（monomorphization），零运行时开销。

第五部分：统一数据模型（Unified Model）

5.1 Rust：刻意没有统一内部模型

Rust 项目没有定义任何"内部统一 Request/Message/Chunk"结构体。整个转换层完全基于 serde_json::Value：

// converter.rs:8
pub fn req_convert(src: &Value, s: Protocol, d: Protocol) -> Result<Value, AppError> {
    match (s, d) {
        (Protocol::Chat, Protocol::Anthropic) => Ok(chat_to_anthropic_req(src)),
        // ...
    }
}
评价说明
优点极致灵活，新字段零改动透传；无序列化→反序列化→重新序列化的开销
缺点没有编译期类型检查；IDE 没有补全；重构时极易漏改

5.2 Go：强类型 + json.RawMessage 折中

Go 项目为每种外部协议定义完整 struct 体系，ResponsesRequest 既是外部协议类型，也充当内部 IR。

pub async fn stream_convert(resp: reqwest::Response, src: Protocol, dst: Protocol, ctx: UsageCtx) -> Result<Response, AppError> {
    let (tx, rx) = tokio::sync::mpsc::channel::<Result<Bytes, std::io::Error>>(64);
    let mut conv = make_converter(src, dst);
    tokio::spawn(async move {
        let mut stream = resp.bytes_stream();
        let mut buf = BytesMut::new();
        while let Some(chunk) = stream.next().await {
            buf.extend_from_slice(&chunk?);
            loop {
                let Some(nl) = buf.iter().position(|b| *b == b'\n') else { break };
                let line = buf.split_to(nl + 1);
                // 解析 SSE 行：event:/data:/空行 → conv.on_event(...)
            }
        }
        // 收尾：on_done()
    });
    Ok(sse_response(Body::from_stream(ReceiverStream::new(rx))))
}

9.2 关键映射差异

usage 的"加减法"哲学差异

Rust（converter.rs:1003）：把 Anthropic 的 input_tokens + cache_creation + cache_read 相加为 Chat 的 prompt_tokens，关注总成本。

Go（responses_to_anthropic.go:103）：把 Responses 的 input_tokens - cache_creation - cached_tokens 相减为 Anthropic 的 input_tokens，关注字段语义对齐。

两者方向相反但都自洽。

thinking signature 的多轮完整性

Rust 完整支持：把 signature 累积在 reasoning_signature 字段（非标准 Chat 扩展），并在 Anthropic 流中发 signature_delta 事件（stream.rs:506）。

Go 项目丢弃入站 thinking（anthropic_to_responses.go:118），仅出站方向把 Responses 的 reasoning.summary 映射回 Anthropic 的 thinking 块——因为 Go 项目主要面向 ChatGPT/Codex 上游，入站 thinking 没有意义。

第十三部分：优缺点与评分

13.1 Rust 项目（ai-aggregs）

优点

  1. 极致简洁：21 个文件覆盖完整协议互转 + 桌面应用 + 配置编辑 + 用量统计
  2. 零开销抽象：Chained<A, B> 泛型单态化，双跳 SSE 转换无运行时成本
  3. 类型安全：尽管用 Value，Protocol enum + match (s, d) 强制覆盖所有组合
  4. 桌面端体验：Tauri 单文件分发 + 系统托盘 + 开机自启
  5. 流式稳定性：流式/非流式降级、usage 嗅探、[DONE] 保证发送
  6. 完整 thinking signature 多轮支持

缺点

  1. 无法接入 Gemini / Bedrock / Vertex
  2. 无 OAuth 支持，无法接入订阅账号
  3. Provider 抽象太弱，无法表达平台特化逻辑
  4. 无 strong-typed model，重构风险高
  5. 无 SaaS 能力（多租户/计费/订阅）

13.2 Go 项目（sub2api）

优点

  1. 强类型协议模型：IDE 补全 + 编译期检查
  2. 完整 SaaS 能力：用户/订阅/支付/促销/兑换/OAuth/限流/计费/审计
  3. 多平台支持：OpenAI / Anthropic / Gemini / Grok / xAI / Vertex / Antigravity
  4. 双路径优化：经 Responses 与直转 Chat 并存
  5. 强大的流式工程：first-output timeout / interval / keepalive / buffer 池 / 失败切换
  6. 完善的 tool calling 支持：namespace / tool_search / custom / server tools 全覆盖
  7. apicompat 包可独立复用

缺点

  1. service 层文件爆炸（200+ 文件）
  2. 没有 Provider interface，扩展新平台需修改多处
  3. 流式状态机无统一抽象，重复代码多
  4. 配置极复杂（数百个 setting）
  5. 依赖庞大（ent ORM 自动生成 5000+ 文件）

第十五部分：迁移建议

15.1 从 Go 项目迁移到 Rust 项目（高价值）

强类型协议 struct（src/protocol/types.rs） 优先级：高

15.2 Rust 项目优于 Go 项目的设计（可作为 Go 演进方向）

StreamConverter trait 统一抽象：替代 Go 的 4 个无接口状态机 优先级：高
Chained<A, B> 组合模式：减少 N² 个状态机到 N 个 优先级：高
BytesMut 零拷贝行缓冲：替代 bufio.Scanner 优先级：中

15.3 共同演进方向

  1. 统一 IR：两者都应有显式 InternalRequest/InternalResponse/InternalMessage 抽象
  2. Provider trait/interface：替代当前的"单 struct + 字段分支"或"多 service + if/else"
  3. 流式工程化：first-output timeout、interval timeout、keepalive、buffer 池应作为标配
  4. 协议适配测试集：建立 round-trip 测试（A→B→A 应等于原值）
  5. 错误标准化：定义跨协议的 error code
